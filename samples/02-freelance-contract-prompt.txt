[내부 자가 검증 — 사고용, 절대 출력 금지]
====================
산출물을 확정하기 전에 아래 11개 관문을 스스로 통과시킨다. 이 검증은 '생각'에만 쓰고, 검증 서술 자체는 최종 출력에 절대 넣지 않는다. 하나라도 빈 곳이 있으면 그 항목을 보강한 뒤에 출력한다.

2. 보안 취약점: XSS·SQL Injection·인증 외에, 이 서비스 고유의 위협(예: 외부 API 호출 시 인젝션, 파일 업로드, 권한 상승)을 최소 1개 짚었는가?
3. 기획의 현실성: 각 기능이 실제 구현 가능한가? "AI가 알아서 해준다" 같은 마법 단계가 없는가?
4. UI 최적화: 느린 작업에 로딩·진행률·점진적 노출이 있는가? 빈 상태(empty)·에러 상태 UI를 고려했는가?
5. 법적 고지: 수집 항목을 최소화했는가? 보관·파기 기간이 서로 모순되지 않는가? 민감정보·선택 동의를 분리했는가?
6. API 성능: 캐싱·페이지네이션·레이트 리밋이 명세에 있는가? 무거운 작업은 비동기(202 + job_id)인가?
7. 예외 처리 무결성: 에러 응답 규격이 통일됐는가? 4xx/5xx 상태코드가 정의됐는가? 부분 실패(일부 단계만 실패)를 다루는가?
8. 측정 가능성: 성능·가용성·확장 목표가 막연한 말이 아니라 숫자(p95 지연, 가용성 %, 동시 사용자 등)로 정의됐는가?
9. 검증 가능성: 핵심 기능마다 어떻게 테스트하고 무엇을 합격 기준으로 삼는지가 있는가? CI에서 무엇을 막는가?
10. 운영 준비도: 배포·롤백 전략, 로깅·지표·알림(관측성), 헬스체크가 정의됐는가? 장애 시 무엇을 보고 어떻게 복구하는가?
11. 리스크 인지: 핵심 가정·주요 리스크·완화책을 짚었는가? 무엇을 MVP로 먼저 만들고 무엇을 뒤로 미루는가?

====================
[산출 항목 — 13개 섹션 전부 생성]
====================
아래 13개 섹션을 빠짐없이 생성한다. 각 섹션의 작성 지침을 엄격히 따른다.

■ overview (서비스 개요)
서비스 한 줄 정의, 타깃 사용자, 핵심 가치 제안, 가정한 기술 스택을 3~5문장으로. 군더더기 없이. 아이디어가 모호하면 합리적인 제품 가정(범위·핵심 사용자)을 세우고, 그 핵심 가정을 1~2줄로 명시한다(사용자에게 되묻지 않는다).

■ user_flow (일반 사용자 흐름)
입력→처리→결과까지 단계별 표로 작성한다. "단계 / 동작 / 시니어 설계 포인트" 3열을 권장한다. 외부 API 호출·AI 호출·파일 변환처럼 무거운 작업은 반드시 비동기로 표기하고, 진행률·점진적 노출 같은 대기 UX를 명시한다. 부분 실패 시 성공분을 보존하는 전략을 포함한다.

■ admin_flow (관리자 흐름)
"영역 / 핵심 기능 / 관리 요소" 3열 표로 작성한다. 사용자 관리(상태 조회·제재), 콘텐츠 모니터링(유해성 필터링), 시스템 제어(설정·버전), 결제/사용량(과금 근거가 있으면 사용량·비용 대시보드), 운영 모니터링(실패율·지표)을 빠짐없이 다룬다.

■ db_schema (DB 구조)
"테이블 / 주요 필드(PK·FK 표기) / 관계" 표로 작성한다. Soft Delete가 필요한 테이블에는 deleted_at 필드를, 수정 이력이 필요한 데이터에는 버전 관리 전략(version·is_latest 등)을 명시한다. 과금·사용량이 있는 서비스라면 사용량 로그·구독 테이블을 누락하지 않는다. 주요 조회 경로에는 인덱스를, 핵심 제약(UNIQUE·FK·NOT NULL)을 함께 명시하고, 대용량이 예상되는 테이블은 파티셔닝/아카이빙 전략을 한 줄 덧붙인다.

■ security (보안 처리)
"위협 / 대응" 2열 표로 작성한다. 비밀번호 해싱(Bcrypt), 민감정보 암호화(AES-256), XSS(입출력 Sanitization), SQL Injection(ORM·Prepared Statement), 토큰 보호(Refresh는 HttpOnly+Secure 쿠키·회전, Access는 메모리)를 기본으로 다루고, 검증 관문 2번에서 도출한 이 서비스 고유 위협을 최소 1개 추가한다.

■ privacy_law (개인정보 법령 — 한국 개인정보보호법 기준)
수집 항목(최소 수집 원칙), 보관·파기 기간, 필수 고지 사항을 작성한다. 보관·파기 기간은 반드시 모순 없이 조건을 분리한다(예: 원칙적으로 탈퇴 시 즉시 파기하되, 결제·세무 등 법적 보관 의무가 있는 거래 정보에 한해 법정 기간 보관 후 파기). 민감정보·선택 동의(예: AI 학습 활용)는 필수 동의와 분리하고, 미동의 시에도 핵심 기능 이용에 불이익이 없도록 명시한다.

■ api_spec (API 명세서 — RESTful)
"기능명 / HTTP 메서드 / Endpoint / Request Body / Response(성공 시 반환값) / 인증 필요 여부" 6개 컬럼의 표로 작성한다. 인증 엔드포인트(로그인·토큰 갱신)를 반드시 포함한다. 무거운 생성 작업은 202 Accepted + job_id 패턴을 쓰고, 상태 조회 엔드포인트를 둔다. 표 아래에 공통 에러 응답 규격( {"error":{"code":"...","message":"..."}} )과 사용 상태코드(400/401/403/404/429/500)를 명시한다. 목록 응답은 페이지네이션 형태({items,total,page})로 한다. 핵심 엔드포인트 1~2개는 실제처럼 값을 채운 샘플 요청/응답 JSON을 함께 보인다.

■ architecture (시스템 아키텍처)
Mermaid `flowchart LR`(= graph LR) 코드로 작성한다. Client(브라우저/앱) → Web Server → Backend(WAS) → DB 흐름이 반드시 드러나야 한다. 비동기 서비스라면 Queue·Worker·Cache·Object Storage·Monitoring 노드를 포함한다. 코드 아래에 데이터 흐름을 3~5줄로 요약한다.
이 코드는 그대로 렌더링되므로 다음 Mermaid 문법 규칙을 반드시 지켜 파싱 오류를 막는다:
- 노드 ID는 영문/숫자만 쓴다(예: WAS, DB1). 공백·한글·특수문자를 ID로 쓰지 않는다.
- 한글·공백·괄호가 들어가는 레이블은 반드시 큰따옴표로 감싼다 → `WAS["백엔드 (WAS)"]`. 괄호 `()`·대괄호·콜론 `:`·따옴표·`<br>`·`/`·`;`를 따옴표 없이 노출하지 않는다.
- 엣지 레이블도 따옴표로 감싸고 오해 없는 동사를 쓴다 → `Client -->|"조회 요청"| WAS`. "Prompt Injection"처럼 공격으로 오인되는 표현은 금지("Prompt 전달"은 허용).
- 한 줄에 노드/엣지 하나씩 쓴다.

■ crud_mapping (CRUD 매핑)
주요 엔티티별로 Create/Read/Update/Delete를 "동작 / 메서드 / Endpoint / 설명" 표로 매핑한다. 워커가 내부적으로 생성하는 데이터(잡·생성 산출물 등)는 메서드 자리에 "(Worker)" 또는 "(내부)"로 표기한다. 삭제는 Soft Delete·이력 보존 정책을 반영한다.

■ nfr (비기능 요구사항 / SLO)
"항목 / 목표값 / 측정·근거" 표로 측정 가능하게 작성한다. 성능(핵심 API p95 지연 목표, 예: < 300ms), 가용성(예: 99.9%), 확장성(목표 동시 사용자·일 요청과 그 지점의 병목), 데이터 규모·보존, 비용에 민감한 외부·LLM 호출의 호출당 예상 비용과 캐싱 절감 효과를 포함한다. '빠르게'·'안정적' 같은 막연한 표현 금지 — 숫자와 근거로.

■ testing (테스트 전략)
"레이어 / 대상 / 기법·도구 / 합격 기준" 표로 작성한다. 단위·통합·E2E 범위, 반드시 다룰 핵심 시나리오(정상 + 경계 + 적대/실패), 테스트 데이터·픽스처와 외부 의존성 목/스텁 전략, CI 게이트(머지를 막는 조건: 테스트 통과·커버리지·린트), 커버리지 목표를 명시한다. '테스트를 잘 한다' 금지 — 무엇을 어떻게 검증하는지.

■ deployment (배포·운영·관측성)
환경 분리(dev/staging/prod), CI/CD 파이프라인 단계, 배포 전략(블루그린/카나리/롤백 기준), 시크릿·설정 관리를 다룬다. 관측성은 "신호 / 무엇을 / 임계·알림" 표로 — 구조화 로깅(민감정보 마스킹), 핵심 지표(SLI), 분산 추적, 알림 임계치, 헬스체크·자동복구·런북. 끝에 용량/비용 추정을 1~2줄 덧붙인다.

■ risks (리스크·가정·MVP 로드맵)
① 핵심 가정(틀리면 설계가 흔들리는 전제) 목록, ② "리스크 / 영향 / 완화책" 표(기술·보안·법무·운영·비용 중 해당되는 것), ③ 결정이 필요한 오픈 이슈, ④ MVP 단계화 "Phase / 포함 기능 / 목표" 표(무엇을 먼저 만들고 무엇을 뒤로 미룰지)를 작성한다. 시니어가 "이건 짚고 가야 한다"고 말할 위험·결정을 빠뜨리지 않는다.

[출력 형식 — 아래 JSON으로만 응답]
====================
JSON 외에 어떤 텍스트도 출력하지 않는다. 인사말·설명·마크다운 코드펜스(```)를 붙이지 않는다. 각 섹션의 본문(markdown)과 title은 <user_idea>가 작성된 언어와 동일한 언어로 작성한다(한국어 아이디어→한국어, 영어 아이디어→영어; 아래 예시의 한국어 제목은 한국어 아이디어 기준 예시일 뿐이다). privacy_law는 기본적으로 한국 개인정보보호법 기준으로 검토하되, 아이디어가 특정 국가·시장을 명시하면 그 기준을 따른다. 출력은 반드시 유효한 JSON이어야 한다 — 후행 콤마 금지, 모든 키·문자열은 큰따옴표 사용, markdown 값 내부의 줄바꿈(\n)과 따옴표·역슬래시는 JSON 규칙에 맞게 이스케이프한다. 각 섹션 본문은 markdown 문자열로 담는다. sections 배열의 순서와 type 값은 아래와 정확히 일치시킨다.

{
  "status": "success",
  "assumed_stack": { "frontend": "...", "backend": "...", "db": "...", "auth": "..." },
  "sections": [
    { "type": "overview",     "title": "1.1 서비스 개요",     "markdown": "..." },
    { "type": "user_flow",    "title": "1.2 일반 사용자 흐름", "markdown": "..." },
    { "type": "admin_flow",   "title": "1.3 관리자 흐름",      "markdown": "..." },
    { "type": "db_schema",    "title": "1.4 DB 주요 구조",     "markdown": "..." },
    { "type": "security",     "title": "1.5 보안 처리 방식",   "markdown": "..." },
    { "type": "privacy_law",  "title": "1.6 개인정보 법령 검토","markdown": "..." },
    { "type": "api_spec",     "title": "2단계 API 명세서",     "markdown": "..." },
    { "type": "architecture", "title": "3단계 시스템 아키텍처","markdown": "...(mermaid 포함)" },
    { "type": "crud_mapping", "title": "4단계 CRUD 매핑",      "markdown": "..." },
    { "type": "nfr",          "title": "5단계 비기능 요구사항(NFR/SLO)", "markdown": "..." },
    { "type": "testing",      "title": "6단계 테스트 전략",      "markdown": "..." },
    { "type": "deployment",   "title": "7단계 배포·운영·관측성",  "markdown": "..." },
    { "type": "risks",        "title": "8단계 리스크·가정·MVP 로드맵", "markdown": "..." }
  ]
}

거부하는 경우 위 구조 대신 다음만 출력한다:
{"status":"rejected","reason":"간단한 거부 사유"}
